# 島人語庫規則 Rules

## Metadata 欄位定義

每筆語料在 `islander_metadata.yaml` 中的格式如下：

```yaml
- id: "00025"
  title: 文章標題
  author: 作者名稱
  date: 2024-01-15          # 原文寫作或發布日期
  created: 2026-04-19       # 入庫日期（當天）
  tags: ["標籤1", "標籤2"]  # 依內容標註，3-8 個為宜
  license: CC-BY-NC-SA-4.0
  filepath: text/分類/編號_標題.md
```

### 格式規範（重要）

- `id` 一律用雙引號包裹
  - 主線編號為五位數字，前方補零，例如 `"00025"`
  - 投稿者若有自訂系列，採用「字母前綴 + 四位數字」格式，例如 `"s0000"`（詳見下方「投稿者自訂系列」）
- `tags` 一律用陣列格式 `["tag1", "tag2"]`，每個 tag 用雙引號
- `date` 和 `created` 使用 `YYYY-MM-DD` 格式，不加引號
- `license` 固定為 `CC-BY-NC-SA-4.0`
- 所有字串值使用雙引號，日期除外
- 主線編號接續目前最後一筆的 id 往下編

### 投稿者自訂系列

部分投稿者會自行採用 `{字母前綴}{四位數字}` 的編號方式（例如 `s0000`、`s0001`…），用以標示該批語料為同一人撰寫。此情況下：

- `id` 保留原本的前綴編號格式（加雙引號）
- `author` 欄位填寫投稿者的姓名或筆名
- 檔名維持原投稿者的命名（`s0000_標題.md`）
- 該投稿者應在 `authors.yaml` 中註冊

這種設計讓每位貢獻者保有各自的編號軌道，同時所有語料仍集中在同一份 metadata 索引中。

### CSV 同步

每次更新 YAML 後，同步更新 `islander_metadata.csv`。CSV 欄位順序：
`id, title, author, date, created, tags, license, filepath`

## 文本檔案格式

每篇語料存為 `.md` 檔，檔名規則：`{id}_{標題}.md`

### 檔案結構：YAML front-matter + 正文

每個 `.md` 檔開頭必須帶有 YAML front-matter（以 `---` 包覆），做為該篇語料 metadata 的**單一真實來源**（single source of truth）。格式如下：

```markdown
---
title: 文章標題
author: 作者名稱
date: 2024-01-15
created: 2026-04-19
tags: ["標籤1", "標籤2"]
license: CC-BY-NC-SA-4.0
source: 原始來源或自創聲明（選用）
notes: 策展註記，如語用價值、文體特徵等（選用）
---

這裡開始是正文。
```

front-matter 後跟空行，再接正文內容。標題中的特殊符號（如書名號《》、引號「」）保留原樣。

### 標點符號訂正規則

收錄時須將正文中的**半形標點**統一替換為全形，保留原文語意，不更動用字。適用於所有來源（FB 貼文、Word 投稿、部落格文章）。

訂正對照表：

| 半形 | 全形 | 說明 |
|------|------|------|
| `"文字"` | `「文字」` | 半形雙引號→書名號式引號 |
| `,` | `，` | 逗號（中文語境） |
| `.` | `。` | 句號（中文語境） |
| `!` | `！` | 驚嘆號 |
| `?` | `？` | 問號 |
| `:` | `：` | 冒號（中文語境） |
| `;` | `；` | 分號 |
| `(文字)` | `（文字）` | 括號（中文語境） |

**例外，以下保留半形**：

- 網址、程式碼、指令中的標點
- 夾雜英文字母或數字的標點（如 `iPhone 15.`、`第 2.3 節`）
- 列舉編號（如 `1.`、`2.`）中的句點

### 中央索引與檔案 front-matter 的關係

`metadata/islander_metadata.yaml` 和 `metadata/islander_metadata.csv` 是由所有檔案的 front-matter 聚合產生的**中央索引**，不是獨立維護的來源。當新增語料或修改既有檔案 front-matter 後，執行：

```
python3 scripts/sync_metadata.py
```

即可重建中央索引，保證兩者一致。修改 metadata 時**應編輯檔案 front-matter**，而非直接改中央 YAML（直接改中央 YAML 會在下次 sync 時被覆蓋）。

## 收錄篩選標準

並非所有文本都適合收錄。在進入整理流程之前，先依以下標準篩選：

### 收錄條件（須同時滿足）

- **字數達 250 字以上**（排除空白字元與換行後計算；差距在 5 字以內的邊界案例從寬認定，超過 5 字則列為待確認）
- **具備文章結構**：有觀點、敘事、論述、描寫，或至少有完整的段落組織

### 進階篩選考量（使用者主觀判斷）

當批量收錄部落格文章時，除了上述基本條件外，還應考慮：

- **異質性**：避免重複主題、相似表述的文章。優先選擇能代表不同視角、不同主題的文本
- **時間範圍**：根據語料庫需求，可優先或排除特定時期的文章（例如 2023 年前、2024 年後等）
- **內容性質**：排除明顯由 AI 生成或 AI 對話紀錄摻雜的文本，即使字數足夠

### 不收錄

- 純日常動態（如「今天好累」「下雨了」）
- 打卡、簽到
- 純轉貼連結，僅附一兩句感想
- 純圖片分享，無實質文字內容
- 活動通知、廣告性質貼文
- AI 生成內容或 AI 對話紀錄

### 模糊地帶

若一篇貼文字數接近 250 字、或介於日常書寫與文章之間（例如一段較長的生活感悟但結構鬆散），標記為「待確認」，列出標題與前幾句內容，交由使用者決定是否收錄。

## 授權驗證機制

所有非使用者本人撰寫的語料，收錄前必須先比對 Google Sheets 授權紀錄。

### 授權表單（Google Sheets）

透過 Google Drive connector 讀取授權回覆試算表。

**試算表檔名**：`島人語庫邀稿與授權表單 (回覆)`

若透過此檔名搜尋找不到，請向使用者確認名稱或網址是否有變更。表單欄位如下：

| 欄位 | 說明 |
|------|------|
| 時間戳記 | 填寫時間 |
| 電子郵件地址 | 投稿者信箱 |
| 姓名或筆名 | 用於比對 metadata 的 `author` 欄位 |
| 授權選項 | 投稿者勾選的授權範圍 |
| 授權類型 | 「上傳檔案」或「部落格文章授權」 |
| 上傳檔案 | 僅接受 .docx 或 .txt（授權類型為「上傳檔案」時使用） |
| 部落格文章網址 | 授權類型為「部落格文章授權」時填寫 |
| 非 AI 聲明 | 投稿者確認語料為本人撰寫且未經 AI 修改 |
| 希望分類 | 投稿者建議的分類（可複選，僅供參考） |
| 聯絡方式 | 選填 |

### 授權比對流程

1. 透過 Google Drive connector 搜尋並開啟試算表「島人語庫邀稿與授權表單 (回覆)」
2. 以**作者姓名/筆名**搜尋是否有對應的授權紀錄
3. 若找到 → 確認授權選項與非 AI 聲明均已勾選 → 通過，繼續收錄流程
4. 若找不到 → **停止收錄**，通知使用者「此作者尚未填寫授權表單」
5. 若授權紀錄存在但非 AI 聲明未勾選 → **停止收錄**，通知使用者

## 單次工作量上限

為避免一次對話消耗過多 token 額度，每次對話中**最多處理 10 篇語料**（含新增、補建、格式修正）。

規則：

- 開始處理前，先統計待處理項目總數，告知使用者
- 若待處理項目超過 10 篇，只處理前 10 篇，其餘留待下次對話
- 處理完 10 篇後，停下來回報進度，提示使用者：「本次已處理 10 篇，如需繼續請開新對話」
- 健檢流程（流程七）不受此限制，因為健檢只產出報告，不實際修改檔案
- 篩選和列出待確認清單也不計入 10 篇上限，只有實際建檔、修改檔案的動作才計入
